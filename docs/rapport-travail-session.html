1 Contexte du Projet

Le HR Talent Portal est une plateforme de recrutement intelligente développée sur Salesforce pour Talan Tunisie. Elle permet de gérer tout le cycle de recrutement : réception des candidatures via Experience Cloud, scoring automatique des CV par IA, planification de tests gaming, passage automatisé des étapes, et publication d'offres d'emploi.

Le tableau de bord recruteur (recruteurDashboard) est le composant central LWC qui fournit une interface moderne pour le suivi des candidatures et la gestion des offres.

4 Corrections & Résolutions de Bugs

4.a — Correction du format DateTime

Résolu Erreur de sérialisation DateTime Salesforce

Problème : Lors de la planification d'un test gaming, Salesforce renvoyait une erreur de sérialisation car le LWC envoyait un string datetime-local (ex: 2026-04-10T14:30) vers un paramètre Apex de type DateTime.

Solution :

  • Changement du type des paramètres Apex de DateTime à String
  • Parsing côté serveur avec DateTime.valueOf(testDateTime.replace('T', ' ') + ':00')
  • Appliqué aux méthodes scheduleGamingTest() et scheduleBatchGamingTests()

Fichiers modifiés : InternalDashboardController.cls

4.b — Correction de l'envoi d'emails aux candidats

Résolu Emails d'invitation non envoyés

Problème : Le code référençait lead.Email directement, mais après conversion du Lead en Opportunity, la variable lead n'existait plus dans le contexte. Aucun email n'était envoyé.

Solution : Implémentation d'un système de fallback à 3 niveaux pour retrouver l'email du candidat :

  1. Niveau 1 — Lead : SELECT Email FROM Lead WHERE ConvertedOpportunityId = :oppId
  2. Niveau 2 — OpportunityContactRole : SELECT Contact.Email FROM OpportunityContactRole
  3. Niveau 3 — Contact : SELECT Email FROM Contact WHERE AccountId = :opp.AccountId

Méthodes corrigées (3) :

  • scheduleGamingTest() — planification individuelle
  • scheduleBatchGamingTests() — planification en lot (avec Maps pré-chargées)
  • updateOpportunityStage() — changement d'étape avec notification
// Fallback email — exemple simplifié
String candidatEmail = null;

// Niveau 1 : Lead converti
List<Lead> leads = [SELECT Email FROM Lead WHERE ConvertedOpportunityId = :oppId LIMIT 1];
if (!leads.isEmpty()) candidatEmail = leads[0].Email;

// Niveau 2 : OpportunityContactRole
if (candidatEmail == null) {
    List<OpportunityContactRole> ocrs = [SELECT Contact.Email
        FROM OpportunityContactRole WHERE OpportunityId = :oppId LIMIT 1];
    if (!ocrs.isEmpty()) candidatEmail = ocrs[0].Contact.Email;
}

// Niveau 3 : Contact via Account
if (candidatEmail == null) {
    List<Contact> contacts = [SELECT Email FROM Contact
        WHERE AccountId = :opp.AccountId LIMIT 1];
    if (!contacts.isEmpty()) candidatEmail = contacts[0].Email;
}

4.c — Nettoyage des enregistrements parasites

Résolu Opportunités parasites dans l'onglet Offres d'emploi

Problème : Deux enregistrements « kemil - Candidature (20%) » et « fatma - Candidature (40%) » apparaissaient dans l'onglet Offres d'emploi car ils avaient le RecordType Job_Posting au lieu de Application.

Diagnostic :

  • RecordType Application (Id: 012fj000004TGblAAG) — actif mais non disponible pour le profil administrateur
  • Impossible de réassigner le RecordType par script Apex

Solution : Suppression des 2 enregistrements parasites. 5 offres d'emploi valides restantes.

4.e — Correction de l'affichage de la description IA

Résolu Description générée par l'IA non visible dans le textarea

Problème : Après clic sur « Générer avec l'IA », la description était bien générée et sauvegardée en base, mais le <textarea> restait vide visuellement dans le modal.

Cause : Dans LWC, un <textarea> natif ne se remet pas à jour via le binding value={offerDescription} après que l'utilisateur a interagi avec le champ. Le DOM property locale prend le dessus sur la propriété tracked.

Solution :

  • Ajout d'une mise à jour directe du DOM après le retour de l'API :
.then(result => {
    this.offerDescription = result;
    this.isGeneratingDesc = false;
    // Force textarea DOM update
    const ta = this.template.querySelector('.form-textarea');
    if (ta) { ta.value = result; }
    this.showToast('Description générée !', 'success');
})
  • Augmentation du nombre de lignes du textarea de 4 à 8 pour mieux afficher les descriptions longues

5 Nouvelle Fonctionnalité : Génération IA des Descriptions d'Offres

🤖 Génération automatique de descriptions d'offres via Groq API

Le recruteur peut saisir une description brouillon (même avec des fautes ou incomplète), puis cliquer sur « Générer avec l'IA » pour obtenir une description professionnelle, structurée et corrigée.

5.3 — Prompt système

Le system prompt instruit le modèle LLaMA à :

  • Agir comme expert en rédaction RH pour Talan Tunisie
  • Structurer la description en 4 sections :
    1. Paragraphe d'introduction (poste + Talan)
    2. Missions principales (liste à puces avec •)
    3. Profil recherché (compétences, expérience)
    4. Avantages (environnement Talan)
  • Corriger toutes les fautes d'orthographe et grammaire
  • Garder un ton professionnel mais dynamique
  • Répondre en texte brut uniquement (pas de HTML ni Markdown)

5.4 — Frontend LWC

HTML : Bouton « Générer avec l'IA » avec spinner de chargement, placé sous le textarea de description

JavaScript :

  • Import de generateJobDescription depuis Apex
  • Variable tracked isGeneratingDesc pour l'état de chargement
  • Handler generateAIDescription() avec validation (titre requis)
  • Mise à jour DOM directe du textarea après réception

CSS : Bouton avec dégradé violet (linear-gradient(135deg, #667eea, #764ba2)), effet hover avec élévation
